pawup 在 OCS2 控制链路中的实现框架图

pawup 的本质是一个接触模式：左前腿 LF 摆动，RF/LH/RH 三腿支撑。 当前工程中，OCS2 显式生成的是摆动腿的 Z 向高度轨迹； XY 落脚轨迹没有被 SwingTrajectoryPlanner 显式规划，而是由 MPC 优化状态再通过正运动学间接得到。

用户手写：gait.info / task.info / reference.info 运行输入：/control_input 或 /cmd_vel 显式足端轨迹：Z 高度样条 隐式足端轨迹：optimized_state -> FK

总流程图

颜色含义：黄色是用户配置/输入，蓝色是运行时数据管理，绿色是轨迹/优化，红色是约束生成。

用户手动写入或触发
控制器运行时搬运数据
MPC / 轨迹生成
约束 / 预计算

1手动参数

gait.info 写入 pawup；task.info 写入摆腿高度参数。

输出：gait 模板、swingHeight、liftOffVelocity、touchDownVelocity

2键盘命令

KeyboardInput 按键 6 发布 command = 6 到 /control_input。

输出：control_inputs_.command = 6

3控制器接收

Ocs2QuadrupedController 订阅 /control_input，写入 CtrlInterfaces。

输入：ROS 消息；输出：控制输入缓存

4GaitManager

command - 2 = 4，选中 gait_list_[4]，也就是 pawup。

输出：插入 GaitSchedule 的 pawup 模板

5TargetManager

根据键盘速度或 /cmd_vel 生成身体目标轨迹 TargetTrajectories。

输出：base 目标状态、默认关节姿态、目标时间

6ReferenceManager

modifyReferences() 从 GaitSchedule 取出当前 MPC 时间窗的 ModeSchedule。

输出：eventTimes + modeSequence

7SwingTrajectoryPlanner

把 RF_LH_RH 展开成 LF 摆动、其余三腿支撑，并生成 LF 的 Z 高度样条。

输出：feetHeightTrajectories_[LF]

8PreComputation

读取 LF 的期望 Z 位置/速度，构造 normalVelocity 约束配置。

输出：eeNormalVelConConfigs_

9SQP MPC

在接触模式、动力学、代价和足端 Z 约束下求解最优状态/输入。

输出：optimized_state、optimized_input、planned_mode

10StateOCS2

evaluatePolicy() 取当前时刻 MPC 策略，交给 WBC。

输入：MPC policy；输出：WBC 期望状态/输入

11WBC

根据 planned_mode 判断 LF 是摆动腿，用优化状态 FK 得到足端 xyz 期望并跟踪。

输出：关节力矩 + 关节位置/速度命令

12硬件接口

命令写入 ros2_control command interfaces，驱动 Gazebo 或真实硬件。

输出：effort / position / velocity / kp / kd

pawup 的关键数据长什么样

gait.info 中的 pawup

pawup
{
  modeSequence
  {
    [0] RF_LH_RH
  }
  switchingTimes
  {
    [0] 0.0
    [1] 2.0
  }
}

RF_LH_RH 表示 RF、LH、RH 支撑；没有写出的 LF 是摆动腿。

Swing Z 轨迹核心公式

liftOff:
  time = swingStartTime
  z    = liftOffHeight
  dz   = liftOffVelocity

midHeight:
  z = max(liftOffHeight, touchDownHeight) + swingHeight

touchDown:
  time = swingFinalTime
  z    = touchDownHeight
  dz   = touchDownVelocity

当前非感知版本中，liftOffHeight 和 touchDownHeight 都来自 terrainHeight = 0.0。

XY 轨迹在哪里

当前实现没有显式 XY 落脚轨迹。

SwingTrajectoryPlanner 只提供 getZpositionConstraint() 和 getZvelocityConstraint()。XY 足端位置来自 OCS2 优化后的 optimized_state，再由 Pinocchio 正运动学算出。WBC 最后跟踪这个 由优化状态得到的足端 xyz，但只有 z 被 swing planner 明确约束。

一句话总结

用户通过 gait.info 写好 pawup 的接触模式，通过按键选择它； GaitManager 把它插入 GaitSchedule； SwitchedModelReferenceManager 把 gait 转成 ModeSchedule； SwingTrajectoryPlanner 根据该接触时序生成 LF 的 Z 向摆腿高度样条； LeggedRobotPreComputation 将这条 Z 轨迹变成 OCS2 约束； MPC 求解后，WBC 根据优化状态和接触模式输出最终关节命令。
